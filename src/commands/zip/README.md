# zip 命令

`ycy zip <directory>` 的设计目标是优先打包可直接交付的前端构建产物，而不是不加判断地把整个目录全部压缩。

这个命令当前刻意采用交互式设计，并不打算在现阶段充当一个安静、面向 CI 的无交互打包器。

## 当前流程

命令当前会按下面这条决策链执行：

1. 解析输入目录。
2. 判断该目录是否像一个 monorepo。
3. 如果是 monorepo，则先让用户选择一个 workspace package。
4. 识别所选 package 的项目类型。
5. 对可能的构建产物目录进行评分，并给出推荐项。
6. 让用户确认或覆盖最终要打包的源目录。
7. 让用户通过 glob 预设进一步收窄要包含的文件。
8. 让用户确认或修改输出 zip 文件名。
9. 压缩文件，并把 zip 写入所选源目录旁边。

## 当前内部结构

当前实现不再把“探测、决策、交互、压缩执行”全部塞在一个文件里，而是拆成四个职责明确的层次：

- `discovery.ts`：负责 workspace/monorepo 探测、项目类型识别、候选产物目录生成与评分
- `engine.ts`：负责把探测结果组织成逐步推进的状态机，决定下一步应该问什么
- `zip.ts`：负责把状态机 step 映射成 `@clack/prompts` 交互，并串起整体流程
- `archive.ts`：负责收集文件、压缩内容、写入 zip 文件

这里有一个刻意的边界：

- 智能引擎会产出 step、候选项、默认值和置信度
- 但它本身不直接依赖 `clack`，避免智能层重新和具体 UI 耦合在一起

这样后续继续扩展规则或改 prompt 体验时，不需要再回到一个超大单文件里做牵一发动全身的修改。

## Monorepo 识别

当命中以下任意一种信号时，命令会把输入目录视为 monorepo：

- `package.json.workspaces`
- `pnpm-workspace.yaml`
- `turbo.json`
- `nx.json`
- `packages/*` 目录布局

一旦识别到 monorepo 信号，命令会扫描 workspace pattern，并先让用户选择一个 package，再继续后面的构建产物识别流程。

## 项目类型识别

当前实现主要聚焦在前端打包场景，尤其是以下几类：

- Vite 项目
- Webpack 项目
- uniapp H5 项目
- 带有 `index.html` 的通用前端目录

这套识别逻辑刻意保持轻量。优先依赖常见配置文件、依赖项和脚本信号，而不是深入解析各种构建配置文件内容。

### 使用到的识别信号

Vite 信号：

- `vite.config.*`
- `vite` 依赖
- 包含 `vite` 的 npm scripts

Webpack 信号：

- `webpack.config.*`
- `webpack` 依赖
- 包含 `webpack` 的 npm scripts

uniapp H5 信号：

- `pages.json`
- `manifest.json`
- `uni.config.*`
- `@dcloudio/vite-plugin-uni` 依赖
- `@dcloudio/uni-app` 依赖
- 包含 `uni` 的 npm scripts

前端兜底信号：

- package 根目录下存在 `index.html`

如果上述信号都没有命中，则命令会回退到 `generic` 模式。

## 构建产物目录推荐

在完成 package 类型识别后，命令会构建一个候选目录列表，并按分数排序。

### 强匹配的已知产物路径

以下目录会获得较高的基础分：

- `dist`
- `build`
- `out`
- `release`
- `public`
- `dist/build/h5`
- `unpackage/dist/build/h5`
- `dist/dev/h5`
- `unpackage/dist/dev/h5`

### 额外评分信号

候选目录还会基于以下启发式条件获得附加分：

- 目录名是否包含 `dist`、`build`、`out`、`release`、`unpackage`、`h5` 等关键词
- 目录中是否存在 `index.html`
- 是否符合已识别工具链的常见产物约定
- 是否位于 `dist/` 或 `unpackage/` 这类常见输出树之下
- 是否需要把 package 根目录本身保留为兜底候选项

内部仍然保留评分理由与命中信号，供状态机和后续测试使用；但 CLI 默认展示会更克制，优先给出置信度和必要提示，而不是把整段评分依据全部直接摊给用户。

## 回退行为

如果命令没有找到明显占优的构建产物目录，它不会立刻失败。

而是会：

1. 先展示一段简短的原因摘要。
2. 回退到一组更小的、可手动选择的候选目录。
3. 仍然允许用户继续选择要打包的目录。

这样即使面对不常见的项目结构，命令仍然可用。

## 文件包含规则

当源目录确定后，命令会继续让用户选择 glob 预设，作为最终包含文件的过滤条件。

当前内置预设：

- `**/*`
- `**/*.html`
- `**/*.js`
- `**/*.css`
- `assets/**/*`
- `static/**/*`

这里的 glob 步骤是第二阶段过滤，只作用于已经选定的源目录内部，不用于替代源目录识别。

## 输出命名与落盘位置

当前行为如下：

- 默认 zip 文件名来自所选 package 名；如果没有 package 名，则回退到 package 目录名
- 用户可以在压缩前手动修改最终 zip 文件名
- zip 文件名中的非法路径字符会被清洗掉
- zip 文件会写入最终选定的源目录中
- 如果已存在同名文件，则直接覆盖

## Zip 内部结构

默认情况下，所选源目录下的内容会直接放在 zip 根目录。

如果用户传入 `--with-dir <dir>`，则所有归档文件都会被包在 zip 内的这个顶层目录下面。

## 当前实现覆盖范围

当前实现主要针对本地交互式使用，并优先服务以下场景：

- 单包前端项目
- monorepo 中的前端 package
- 常见的 Vite 和 Webpack 构建产物
- uniapp H5 输出目录

它目前还不是一个完全确定性、面向 CI 的打包工具。

## 有意为之的取舍

- 优先交互确认，而不是静默自动化。
- 优先轻量识别，而不是解析所有可能的构建配置格式。
- 优先保留内部可解释性，但默认输出更克制，避免提示信息过载。
- 当证据足够时，优先打包构建产物，而不是源码目录。

## 后续可扩展方向

如果后续继续扩展这个命令，比较自然的下一步包括：

- 解析部分工具链配置，以更准确地发现自定义输出目录
- 增加显式的预览或 explain 模式
- 增加更适合脚本和 CI 的非交互路径
- 支持超出前端构建产物之外的更广泛打包规则
